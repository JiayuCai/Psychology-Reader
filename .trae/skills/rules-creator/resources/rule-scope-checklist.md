# Rule Scope Checklist

在起草规则前，先用这份清单判断“这条要求该写成什么”。

## 第一步：是否真的适合写成规则？

- [ ] 这是长期重复出现的要求，而不是一次性说明
- [ ] 这是稳定约束，而不是随任务变化的临时偏好
- [ ] 用户希望以后默认遵守，而不是本次对话临时采用

如果以上多数不满足：

- 不建议写成 Rule
- 优先改为一次性 prompt 或任务内说明

## 第二步：应写成哪一类规则？

### 全局规则

满足以下任一情况，优先考虑全局规则：

- [ ] 属于用户的长期沟通偏好
- [ ] 属于用户的整体输出偏好
- [ ] 希望在所有项目中一致生效

典型例子：

- 回答风格偏简洁或偏结构化
- 默认使用某种语言
- 默认先给框架再展开细节

### 项目规则

满足以下任一情况，优先考虑项目规则：

- [ ] 仅对当前项目有效
- [ ] 属于当前项目的技术栈、代码风格或模块约束
- [ ] 该约束不应污染其他项目

典型例子：

- 命名规范
- 测试框架偏好
- API 使用限制

### 子目录规则

满足以下任一情况，优先考虑子目录规则：

- [ ] 只针对某个模块、子系统或子目录
- [ ] 同一项目内不同目录使用不同技术栈
- [ ] 根目录规则会对其他模块造成干扰

典型例子：

- `frontend-module/` 使用 React 约束
- `backend-module/` 使用 API 设计约束

## 第三步：项目规则应如何生效？

### 始终生效

- [ ] 该规则在当前项目的所有相关对话中都必须生效
- [ ] 不依赖具体文件类型或上下文判断

### 指定文件生效

- [ ] 该规则只针对特定文件类型、路径或目录
- [ ] 可以用 `globs` 明确表达作用范围

### 智能生效

- [ ] 该规则只在某类任务场景下相关
- [ ] 更适合用一段 `description` 描述适用时机

### 手动触发生效

- [ ] 该规则不应默认生效
- [ ] 仅当用户显式使用 `#Rule` 时才应用

## 第四步：是否需要拆分？

- [ ] 是否把沟通偏好、代码风格、技术栈限制混在了一条规则里
- [ ] 是否同时覆盖多个不相关目录
- [ ] 是否同时包含“总原则”和“模块专属例外”

如果以上任一为是：

- 建议拆成多条规则

## 第五步：质量自检

- [ ] 规则内容是否可执行
- [ ] 是否能用“应/优先/禁止”重写
- [ ] 是否避免空泛表达
- [ ] 是否说明了作用范围
- [ ] 是否选择了正确的生效方式
