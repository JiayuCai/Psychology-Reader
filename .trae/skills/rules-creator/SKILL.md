# Rules Creator

## 作用

该 Skill 用于按照 TRAE 官方 Rules 机制，帮助用户把个人偏好、团队规范、项目约束或模块约束，转写为结构清晰、作用域明确、可长期复用的规则文件。

它不只是“帮用户写一段规则文本”，而是帮助用户判断：

- 这条要求是否适合写成规则
- 应写成全局规则还是项目规则
- 项目规则应使用哪种生效方式
- 规则应放在项目根目录还是子目录
- 是否需要拆成多条规则而不是混写在一起

## 何时使用

当出现以下情况时，应优先调用本 Skill：

- 用户想创建 TRAE 的全局规则或项目规则
- 用户想把沟通偏好、代码风格、命名规范、框架约束整理为规则
- 用户想判断某条规则应设为始终生效、指定文件生效、智能生效还是手动触发
- 用户想为某个子目录或模块创建专属规则
- 用户已有一段自然语言规范，希望整理成符合 TRAE 规则机制的文件
- 用户想重写现有规则，使其边界更清晰、冲突更少、命中更稳定

## 何时不要使用

以下情况不要调用本 Skill：

- 用户只是想临时完成一次性任务，而不是沉淀为长期规则
- 用户要创建的是 Skill，而不是 Rule
- 用户要接入的是 MCP Server，而不是规则文件
- 用户给出的要求更适合作为单次 prompt，而不是长期生效的约束
- 用户只是想了解 Rules 功能入口或界面操作，不需要起草规则内容

## 输入清单

开始起草前，优先确认以下信息：

- 这条约束属于个人偏好、团队规范、项目约束还是模块专属约束
- 规则应在所有项目生效，还是仅在当前项目生效
- 若仅在当前项目生效，是否只针对某类文件或某个子目录
- 用户希望这条规则如何触发：
  - 始终生效
  - 指定文件生效
  - 智能生效
  - 手动触发生效
- 规则的核心内容是什么
- 是否已有旧规则需要重写、拆分或合并
- 是否需要附带规则名称、建议文件路径和 frontmatter 配置

如果这些信息不完整，应先澄清，不要直接生成模糊规则。

## 默认执行流程

1. 先判断用户需求是否适合写成 Rule，而不是 Skill、MCP 或一次性 prompt。
2. 判断规则作用域：
   - 个人长期偏好 -> 全局规则
   - 当前项目长期约束 -> 项目规则
   - 某个模块或子目录专属约束 -> 子目录规则
3. 判断项目规则的生效方式：
   - 对所有对话都应生效 -> 始终生效
   - 只对特定文件类型或路径生效 -> 指定文件生效
   - 需由 AI 根据场景判断 -> 智能生效
   - 仅在用户显式引用时生效 -> 手动触发生效
4. 根据生效方式生成建议配置：
   - `alwaysApply`
   - `globs`
   - `description`
   - 是否建议通过 `#Rule` 手动引用
5. 起草规则正文：
   - 用明确约束表达，不写空泛口号
   - 尽量写“应/不应/优先/禁止”这类可执行表述
   - 避免把多条无关要求混成一条规则
6. 如果规则较复杂，拆成多条：
   - 风格类
   - 技术栈类
   - 目录/模块类
   - 交互偏好类
7. 给出最终交付建议：
   - 规则名称
   - 建议文件路径
   - 生效方式
   - 可直接粘贴的 Markdown 内容

## 输出要求

默认输出至少应包含：

- 规则类型判断：
  - 全局规则 / 项目规则 / 子目录规则
- 生效方式判断：
  - 始终生效 / 指定文件生效 / 智能生效 / 手动触发
- 建议文件位置
- 可直接使用的规则内容草案
- 若适用，补充：
  - `alwaysApply`
  - `globs`
  - `description`
  - `#Rule` 使用建议

如果用户一次提出多类约束，应优先按“规则文件拆分方案”输出，而不是全部塞进一个文件。

## 失败策略

- 如果用户需求更适合 Skill：明确指出这是 Skill 场景，并建议改用 `skill-creator` 流程
- 如果用户需求更适合一次性 prompt：说明不建议写成长期规则，并给出原因
- 如果用户同时混入多个层级的约束：先拆分为全局规则、项目规则、子目录规则三个候选
- 如果用户无法判断生效方式：先给出推荐方案，并说明为什么
- 如果规则内容过于抽象，例如“写得更好”“更专业”：先追问可执行标准，再起草
- 如果用户要求一条规则同时覆盖过多文件、模块和行为：建议拆分，避免命中不稳定
