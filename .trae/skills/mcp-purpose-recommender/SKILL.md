# MCP Purpose Recommender

## 作用

该 Skill 用于根据用户当前目标与用途，快速推荐最适合的 MCP 候选，而不是做泛化的生态盘点或过度核验。

它优先服务“我现在要解决什么问题”，例如：

- 浏览器网页内容抓取
- PDF 解析
- 学术检索
- 搜索增强
- 设计稿转代码
- 自动化测试

默认采用“用途优先、Awesome 优先、按需补充”的检索策略，避免把任务自动改写为“找官方来源”“找网页目录”或“做完整选型报告”。

## 何时使用

当出现以下情况时，应优先调用本 Skill：

- 用户想找一些“好用的 MCP”
- 用户已说明具体任务，希望根据用途推荐 MCP
- 用户想快速得到一组值得试装的 MCP shortlist
- 用户想知道某类需求通常该搜哪几类 MCP
- 用户希望减少冗余核验，优先以可装可用为目标进行推荐

## 何时不要使用

以下情况不要调用本 Skill：

- 用户要的是 MCP 协议说明、规范解读或官方文档导航
- 用户要的是完整的技术选型报告，而不是快速推荐候选
- 用户已经明确指定了某个 MCP，只需要安装或配置
- 用户要构建自己的 MCP Server，而不是寻找现成 MCP
- 用户要的是 TRAE 规则、Skill 或普通 prompt，而不是 MCP 推荐

## 核心原则

- 目标优先：先围绕用户用途构建候选，不默认按“官方/市场/网页目录”分类
- Awesome 优先：默认先从 `awesome-mcp-servers` 建候选池
- 社区补充：当 `awesome-mcp-servers` 信息不足时，优先补看 Glama 这类社区目录
- 最小验证：如果 `awesome-mcp-servers` 已能提供清晰候选与项目链接，则不强制补查官方 registry 或 market
- 按需补充：只有当信息不足、安装不明、维护状态可疑、或用户明确要求时，才补看其他来源
- 少而准：优先输出 3-5 个最值得试的 MCP，而不是堆很多名字
- 明确理由：每个推荐都要说明“为什么适合这个用途”

## 输入清单

开始推荐前，优先确认以下信息：

- 用户当前想解决的任务是什么
- 用户更重视什么：
  - 快速能用
  - 直接可装
  - 功能最强
  - 维护稳定
- 用户当前使用的平台是否重要，例如 TRAE
- 是否要求优先推荐可直接在 TRAE 中安装的 MCP
- 是否接受需要 API Key 或远程服务的 MCP
- 是否已有用户点名想看的来源，例如：
  - `awesome-mcp-servers`
  - TRAE MCP 市场
  - Glama
  - MCP Market

如果用户已明确说明目标与用途，不要再把任务改写为更窄的“找官方来源”或“找网页目录”问题。

## 默认执行流程

1. 提炼用户用途
   - 用一句话概括用户当前要解决的问题
   - 例如：网页抓取、浏览器自动化、正文抽取、搜索增强

2. 将用途映射为 MCP 类型
   - 浏览器自动化
   - 网页正文抽取
   - 搜索与检索增强
   - 文档/PDF 处理
   - API 集成
   - 设计/开发自动化

3. 默认优先检查 `awesome-mcp-servers`
   - 根据用途查对应分类或关键词
   - 构建一个初始候选池

4. 对候选做轻量筛选
   - 是否明显符合用途
   - 是否有清晰项目主页或安装说明
   - 是否看起来仍在维护
   - 是否适合用户当前平台或工作流

5. 当 `awesome-mcp-servers` 信息不足时，优先补看社区目录
   - 优先使用 Glama 或类似社区目录补充候选、比较维护度和发现替代项
   - 适用情形包括：
     - awesome 中候选过少
     - 条目描述过短
     - 难以判断谁更成熟
     - 想找同类替代方案
   - 这一阶段的目标是补充“更好用的候选”，而不是做官方核验

6. 再看落地安装入口
   - 如果用户关心是否可直接装进 TRAE，则检查 TRAE MCP 市场
   - 如果市场里能直接搜到同名项，优先建议先试装

7. 仅在必要时补看官方来源
   - 如果用户明确要求官方出处、需要核验真实性，或前述来源信息冲突，再补看官方 registry 或文档

8. 输出 shortlist
   - 推荐 3-5 个候选
   - 每个候选说明：
     - 适合什么用途
     - 为什么推荐
     - 可能的限制
     - 是否建议优先试装

9. 如有需要，补一个检索方法
   - 告诉用户应优先在什么来源里搜什么关键词
   - 不默认展开成冗长调研

## 输出要求

默认输出应包含：

- 用户用途的一句话重述
- 推荐的 MCP 类型
- 3-5 个候选 MCP
- 每个候选的简短推荐理由
- 是否建议先在 TRAE 市场中搜索同名项
- 若适用，补一个“最省时间的检索顺序”

如果用户要求的是“快速开始”，优先给 shortlist，不展开过多背景。

如果用户要求的是“怎么搜”，优先输出：

- 从哪个来源开始
- 搜哪些关键词
- 什么时候需要切换到其他来源

## 失败策略

- 如果用户目的不清：先澄清用途，而不是直接列工具
- 如果候选太多：收缩为 3-5 个最适合当前用途的 MCP
- 如果 `awesome-mcp-servers` 信息已足够：不要强行补官方和 market
- 如果用户明确要求“只看官方”或“只看 TRAE 市场”：遵从用户要求
- 如果某个候选看起来强但安装或维护信息不清：明确标注不确定性
- 如果用户真正需要的是安装指导：停止泛推荐，切换到安装落地流程

## 推荐来源优先级

默认优先级如下：

1. `awesome-mcp-servers`
2. 社区目录（如 Glama）
3. TRAE MCP 市场
4. 官方 registry 或文档

注意：

- 该顺序服务于“快速找到好用且可装的 MCP”
- 社区目录用于补充候选、比较成熟度和发现替代项
- TRAE MCP 市场用于确认是否能直接落地安装
- 官方来源仅在需要核验时再查看
- 若用户已指定来源，应优先使用用户指定来源

## 典型触发语句

- “帮我找一些好用的 MCP”
- “我这个需求适合装什么 MCP”
- “做网页内容抓取该找哪类 MCP”
- “帮我按用途推荐几个 MCP”
- “怎么高效搜适合当前任务的 MCP”
